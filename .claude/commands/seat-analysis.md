Claude Team プランのシート最適化分析を実行してください。手順:

## 0. 対象の決定

- 引数 `$ARGUMENTS` を解釈する: `YYYY-MM` 形式なら対象月、`preview` と日数
  （例: `preview 10` / `preview 10日`）があれば速報モード、それ以外の語は組織名
  （`input/` 直下のディレクトリ名）として扱う。指定が無ければ **全組織 × 最新月**
- **速報モードの場合**は以降の手順の代わりに:
  1. `uv run seat-analyzer analyze --preview --days <日数> [--org <組織名>] [--month YYYY-MM]` を実行
  2. `reports/<組織名>/<月>/preview.md` の「## 考察」を執筆する。観点:
     遊休候補への声かけ・オンボーディング確認、⚠️超過済ユーザの妥当性、
     Standard候補の暫定リスト（確定判断は正式分析に委ねる旨を明記）、
     次の正式分析までに観察すべきポイント
  3. 一次判断の内訳と考察の要点をユーザに報告して終了

## 1. 入力ファイルの確認

- `input/<組織名>/{spend,members,code-analytics}/` の中身を組織ごとに確認する
- 対象月のスペンドレポートとメンバー一覧が揃っているか確認する。
  欠けている組織があれば README の月次運用手順を案内し、その組織を除外するか
  中断するかをユーザに確認する
- ファイルのヘッダを確認し、`config.yaml > columns` のエイリアスで解決できない
  カラム名があれば config.yaml にエイリアスを追記する

## 2. 分析の実行

```sh
uv run seat-analyzer analyze [--month YYYY-MM] [--org <組織名>]
```

`--org` 省略で全組織を一括分析。エラーになった場合はメッセージに従い修正
（カラムマッピング追記など）して再実行する。

## 3. 結果の検証（組織ごと）

- CLI の警告を確認する。特に:
  - **spend突合の乖離警告** → `config.yaml > model_prices` の単価が最新か、
    spend列の意味（実課金 vs API等価見積り）を確認し、README に解釈を記録する
  - **シート不明ユーザ** → members ファイルの更新漏れの可能性を指摘する
    （「対象外（シート未割当）」は意図的な未割当のため指摘不要）
  - **スキップされた組織** → 対象月のエクスポート漏れ。手順を案内する
- `reports/<組織名>/<月>/recommendations.csv` を読み、外れ値（極端な API換算コスト、
  前月からの急変）がないか確認する

## 4. 考察の執筆（組織ごと）

各組織の `reports/<組織名>/<月>/report.md` の「## 考察」セクションを以下の観点で書き換える:

- **変更推奨の妥当性**: 各ユーザについて、活用度（PR数・行数）と突き合わせて
  「本当に使っていないのか、オンボーディング不足なのか」を評価する。
  members-info の備考（ヒアリング済み・休職中等）があるユーザは、その事情を
  考察に反映し、同じ確認を繰り返さない。部署別・チーム別サマリがあれば
  部署間・チーム間の利用密度・削減余地の偏りにも触れる
- **前月からの変化**: 過去月のレポートがあれば、コスト・推奨の変化点を要約する
- **上限到達疑いユーザ**: 従量課金の有効化状況と合わせた解釈を書く
  （無効なら「需要が観測上限で切られている可能性」を明記）
- **次のアクション**: シート変更の実施手順（管理画面での変更、本人への事前連絡）と、
  次月に見るべきポイントを箇条書きにする

複数組織を分析した場合は、組織間の比較（利用密度・削減余地の偏り）にも触れる。
考察は事実（数値）と解釈を分けて書き、数値はすべて CSV / report.md から引用すること。

## 5. 完了報告

削減見込み額・変更推奨ユーザ・注意点を組織ごとに簡潔にまとめ、複数組織なら
`reports/summary/<月>.md` の横断サマリ（全体合計）も添えてユーザに報告する。
`reports/<組織名>/<月>/dashboard.html` を経営層共有用として案内する。
